# S-06.04: Block Palette Filtering (`availableBlocks`)

**Epic:** E-06
**Size:** XS
**Dependencies:** S-06.03 (`BlockEditor.tsx`)

## Skills to Load

- `content-pack-system` — `availableBlocks` field in `phase1.challenge` / `phase2.challenge` / `phase3`
- `react-component-structure` — extending existing component props

## What

The lesson JSON declares which blocks are available to the player per phase (`availableBlocks: string[]`). The `BlockEditor` already receives this array and passes it to `buildToolbox`. This story formalises the toolbox builder to group blocks by their `category` field (from `BlockDef`), so the toolbox renders with labeled category headers rather than a flat list.

This is a small but important UX story — a toolbox with category headers ("Movement", "Drawing") is far easier to navigate than an undifferentiated pile of blocks.

## Tasks

1. Update `buildToolbox` in `BlockEditor.tsx` to accept both `blockIds: string[]` and `blockDefs: BlockDef[]`:
   ```ts
   function buildToolbox(blockIds: string[], blockDefs: BlockDef[]): object {
     // Group available blocks by category
     const categories = new Map<string, string[]>();
     for (const id of blockIds) {
       const def = blockDefs.find((b) => b.id === id);
       if (def === undefined) continue;
       const cat = def.category;
       if (!categories.has(cat)) categories.set(cat, []);
       categories.get(cat)!.push(id);
     }

     return {
       kind: 'categoryToolbox',
       contents: Array.from(categories.entries()).map(([name, ids]) => ({
         kind: 'category',
         name,
         contents: ids.map((id) => ({ kind: 'block', type: id })),
       })),
     };
   }
   ```

2. Pass `blockDefs` into `buildToolbox` when calling it inside the `useEffect` in `BlockEditor.tsx`.

3. If only one category is present (common in early lessons), the toolbox still renders with a single labeled category — this is correct Blockly behaviour and looks fine.

4. Blocks whose IDs are not in `blockDefs` (registration mismatch) are silently skipped with a `console.warn`. Document this in a comment.
