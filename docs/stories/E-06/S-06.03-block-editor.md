**Epic:** E-06
**Size:** S
**Dependencies:** S-06.02 (`blockRegistry.ts`), S-06.01 (`blocklyTheme.ts`)

## Skills to Load

- `react-component-structure` ŌĆö `useRef`, `useEffect` lifecycle, named export
- `naming-conventions` ŌĆö `BlockEditor.tsx` lives in `src/editor/`

## What

`BlockEditor` is a React component that manages the Blockly workspace lifecycle. It creates a `<div>` ref that Blockly injects into on mount, and calls `workspace.dispose()` on unmount to prevent memory leaks. The workspace configuration (theme, toolbox, scroll settings) is applied at injection time.

`BlockEditor` is a thin wrapper ŌĆö it does not own the run button (S-06.05) or serialization (S-06.06). Those are added incrementally. This story establishes the inject/dispose lifecycle and the workspace reference.

## Tasks

1. Create `src/editor/BlockEditor.tsx`:
   ```tsx
   import { useEffect, useRef } from 'react';
   import * as Blockly from 'blockly/core';
   import { codeQuestTheme } from './blocklyTheme';
   import { registerBlocks } from './blockRegistry';
   import type { BlockDef } from '@/types/content';

   export interface BlockEditorProps {
     blockDefs: BlockDef[];
     availableBlocks: string[];
     onCodeGenerated: (code: string) => void;
     readOnly?: boolean;
     initialState?: Record<string, unknown>;
   }

   export function BlockEditor({
     blockDefs,
     availableBlocks,
     onCodeGenerated,
     readOnly = false,
     initialState,
   }: BlockEditorProps): React.ReactElement {
     const containerRef = useRef<HTMLDivElement>(null);
     const workspaceRef = useRef<Blockly.WorkspaceSvg | null>(null);

     useEffect(() => {
       if (containerRef.current === null) return;

       // Register all block types from the lesson content pack
       registerBlocks(blockDefs);

       // Build toolbox from availableBlocks (S-06.04 will expand this)
       const toolbox = buildToolbox(availableBlocks);

       // Inject workspace
       const workspace = Blockly.inject(containerRef.current, {
         theme: codeQuestTheme,
         toolbox,
         readOnly,
         scrollbars: true,
         trashcan: true,
         zoom: { controls: true, wheel: true, startScale: 0.9 },
       });

       workspaceRef.current = workspace;

       // Restore saved state if provided
       if (initialState !== undefined) {
         Blockly.serialization.workspaces.load(initialState, workspace);
       }

       return () => {
         workspace.dispose();
         workspaceRef.current = null;
       };
     }, []); // Intentionally empty ŌĆö workspace lifecycle is mount/unmount only

     return (
       <div className="flex flex-col w-full h-full">
         <div ref={containerRef} className="flex-1 w-full" />
       </div>
     );
   }
   ```

2. `buildToolbox` is a local helper (in the same file) that converts an array of block type IDs to a Blockly toolbox definition object:
   ```ts
   function buildToolbox(blockIds: string[]): object {
     return {
       kind: 'flyoutToolbox',
       contents: blockIds.map((id) => ({ kind: 'block', type: id })),
     };
   }
   ```
   This is the minimal toolbox format. S-06.04 will extend it with category grouping.

3. The `useEffect` dependency array is intentionally empty (`[]`) ŌĆö the workspace is injected once on mount and cleaned up on unmount. The `blockDefs` and `availableBlocks` props are not expected to change during the component's lifetime. If they do change (e.g. phase transition), the parent should unmount and remount `BlockEditor` using a `key` prop.

4. Do not put `workspaceRef` in React state. The workspace is a mutable imperative object ŌĆö mutating it does not need to trigger re-renders. Store it in a ref.

5. The `onCodeGenerated` callback is wired in S-06.05 (run button). Pass it through to keep the interface stable.
