# S-04.06: Node Click Handler

**Epic:** E-04
**Size:** XS
**Dependencies:** S-04.05 (MapCanvas wires the handler)

- `testing-strategy` — **required** — outcome-focused validation before writing any test

- `react-component-structure` — prop drilling vs context

## What

When the player clicks an `available` node on the map, the platform needs to start the lesson for that node. This story defines the callback contract that connects `MapCanvas` to the rest of the app.

The click is already handled at the `MapNode` level (S-04.01 fires `onClick` only when `state === 'available'`). This story is about confirming the callback prop signature is correct and verifying that `MapCanvas` correctly propagates the selected `nodeId` up to whoever holds navigation state.

For now the callback just needs to be accepted as a prop on `MapCanvas` and called with the clicked node's ID. The actual navigation (opening a lesson, transitioning to Phase 1) is wired in E-11. `MapScreen` (S-04.08) passes a stub handler until then.

## Tasks

1. Confirm that `MapCanvas` already accepts `onNodeSelect: (nodeId: string) => void` as a prop (defined in S-04.05). No new file is needed.

2. Verify the data flow:
   - User clicks a `MapNode` button
   - `MapNode` fires its `onClick` prop (only when `state === 'available'`)
   - `MapCanvas` passes `() => onNodeSelect(node.id)` as the `onClick` prop to each `MapNode`
   - `MapCanvas`'s `onNodeSelect` prop bubbles up to `MapScreen` and eventually to `App.tsx`

3. In `MapScreen` (S-04.08), wire a stub handler for now:
   ```tsx
   function handleNodeSelect(nodeId: string): void {
     // TODO(E-11): trigger lesson start for nodeId
     console.warn('Node selected — lesson start not yet implemented:', nodeId);
   }
   ```
   Using `console.warn` is intentional — it satisfies the Hard Rule (no `console.log`) and leaves a visible signal during development.

4. Do not use `console.log` — the linter enforces `warn`/`error` only (see Hard Rules).
