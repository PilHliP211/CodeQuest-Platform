# S-07.02: Code Label Inside Block Body

**Epic:** E-07
**Size:** S
**Dependencies:** S-07.01 (phase prop wired)

## Skills to Load

- `react-component-structure` — where Blockly customization code lives relative to `BlockEditor.tsx`
- `typescript-standards` — no `any`; type Blockly workspace and block objects using the `blockly` package's own types
- `security-rules` — the `code` field from `BlockDef` is inserted as a text label, never as HTML or evaluated code
- `content-pack-system` — `BlockDef.code` is the source of truth for the label text

## What

When `phase === 2`, each block in the Blockly workspace shows its `code` field (from `BlockDef` in the lesson JSON) as a secondary text label rendered inside the block body. The label uses a monospace font, small size, and 60% opacity so it reads as supplemental information rather than the primary block label. The block remains fully functional: draggable, connectable, and executable.

Blockly supports adding extra fields to a block definition. The cleanest approach is to modify `blockRegistry.ts` to inject an additional `field_label` when `phase === 2` at the time blocks are registered. If the Blockly renderer does not support per-field opacity directly, apply opacity via a custom CSS class injected into the Blockly SVG DOM after render.

## Tasks

1. In `blockRegistry.ts`, accept an optional `showCode: boolean` parameter. When `true`, inject an extra `Blockly.FieldLabel` beneath the block's primary label:
   ```ts
   export function registerBlocks(blocks: BlockDef[], showCode = false): void {
     blocks.forEach((def) => {
       const blocklyDef = { ...def.blocklyDef };
       if (showCode) {
         // Append a second message line with the code field
         blocklyDef.message1 = def.code;
         blocklyDef.args1 = [];
       }
       Blockly.Blocks[def.blocklyDef.type] = {
         init(this: Blockly.Block) {
           this.jsonInit(blocklyDef);
         },
       };
     });
   }
   ```

2. In `BlockEditor.tsx`, pass `showCode={phase === 2}` to `registerBlocks`.

3. Style the secondary label:
   - Font family: `monospace` (do not use Press Start 2P — it is too wide at small sizes)
   - Font size: `10px` (or Tailwind equivalent if applied via wrapper class)
   - Opacity: `0.6`
   - Apply via a targeted CSS rule scoped to `.blocklyFlyout` and `.blocklyWorkspace` SVG text elements that carry a `data-code-label` attribute, OR inject a `<style>` tag into the Blockly iframe/shadow DOM if required.

4. Verify the code label text is sourced from `BlockDef.code` (the content pack field), not reconstructed or hardcoded.

5. Run a smoke test: confirm blocks still drag, connect, and produce the correct code string when the workspace is serialized.

## Done When

- [ ] When `phase === 2`, every block in the workspace displays its `code` field as a second label
- [ ] Label is monospace, `10px`, 60% opacity
- [ ] Label text matches `BlockDef.code` from the content pack exactly
- [ ] Blocks remain draggable, connectable, and executable in Phase 2
- [ ] Phase 1 blocks show no code label
- [ ] `npm run lint` and `npx tsc --noEmit` pass
