# S-03.06: Content Load Error Gate

**Epic:** E-03
**Size:** XS
**Dependencies:** S-03.05 (useContent hook)

## Skills to Load

- `testing-strategy` — **required** — outcome-focused validation before writing any test
- `component-tests` — Testing Library behavior tests for the component added here
- `react-component-structure` — conditional rendering at the app root
- `react-context-pattern` — reading `ContentErrorContext` to drive gate decision
- `accessibility` — full-screen error messages must be screen-reader accessible
- `error-handling` — never show a blank screen on fatal errors

## What

If the content pack fails to load (validation error, missing file, corrupt JSON), the platform must show a clear full-screen error message rather than a blank page or a runtime exception that crashes the React tree. This gate lives in `App.tsx` and is analogous to the profile gate from S-02.10 — except instead of redirecting to a setup screen, it shows an unrecoverable error with a helpful message.

The gate reads from `ContentErrorContext` (always defined, even when the course is null) to check for an error before attempting to render anything that calls `useContent()`.

## Tasks

1. Update `App.tsx` to read the content error before rendering the main app tree:

   ```tsx
   import { useContext } from 'react';
   import { useProfile } from '@/engine/ProfileContext';
   import { ContentErrorContext } from '@/engine/ContentContext';
   import { NameEntryScreen } from '@/components/Profile/NameEntryScreen';

   export function App(): React.ReactElement {
     const contentError = useContext(ContentErrorContext);
     const { profile } = useProfile();

     if (contentError !== null) {
       return (
         <main
           role="alert"
           aria-live="assertive"
           className="flex min-h-screen flex-col items-center justify-center bg-gray-900 p-8 text-center"
         >
           <h1 className="mb-4 font-['Press_Start_2P'] text-xl text-red-400">
             Content Pack Error
           </h1>
           <p className="mb-6 max-w-lg font-['Press_Start_2P'] text-sm leading-relaxed text-gray-200">
             The game content failed to load. Please check the browser console for details.
           </p>
           <pre className="max-w-lg overflow-auto rounded bg-gray-800 p-4 text-left text-xs text-yellow-300">
             {contentError}
           </pre>
         </main>
       );
     }

     if (profile === null) {
       return <NameEntryScreen />;
     }

     return <main>{/* main app placeholder */}</main>;
   }
   ```

2. The content error check must come **before** any call to `useContent()` (directly or via child components). The gate ensures that nothing below it attempts to use an undefined `Course`.

3. The error UI uses `role="alert"` and `aria-live="assertive"` so screen readers announce the error immediately.

4. The error message from `ContentErrorContext` (the technical validation message) is shown verbatim in a `<pre>` block. This is intentional — developers need the raw error to diagnose content pack issues. The friendly heading is for end users (or parents) who encounter a broken build.

5. No "retry" button. Content packs are static in v1 — if it failed, a page reload will produce the same result. A broken content pack requires a fix and redeploy.
