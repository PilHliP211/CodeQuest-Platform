# S-09.05: Friendly Error Messages

**Epic:** E-09
**Size:** S
**Dependencies:** S-09.04 (`interpreter.ts`)

## Skills to Load

- `testing-strategy` — **required** — outcome-focused validation before writing any test
- `unit-tests` — Vitest AAA+O pattern for the pure logic added here
- `error-handling` — typed error surface, no raw stack traces exposed
- `typescript-standards` — `PlayerFriendlyError` type, exhaustive mapping

## What

`PlayerFriendlyError` is already defined in S-09.04, but this story hardens the error message text to be genuinely kid-friendly (ages 8-12). Stack traces, internal module names, and technical jargon should never reach the UI. Each error category gets a specific, actionable message.

This story also adds a `formatErrorForDisplay` utility that turns a `PlayerFriendlyError` into a display-ready string with optional line number prefix.

## Tasks

1. Add `formatErrorForDisplay` to `interpreter.ts` (or a sibling `errors.ts`):
   ```ts
   export function formatErrorForDisplay(error: PlayerFriendlyError): string {
     if (error.line !== undefined) {
       return `Line ${error.line}: ${error.message}`;
     }
     return error.message;
   }
   ```

2. Audit all error messages in `sandbox.ts` and `interpreter.ts` and rewrite any that sound technical. Target language: clear enough for a 10-year-old to understand what to fix. Examples:

   | Technical message | Kid-friendly replacement |
   |---|---|
   | `Unexpected token '('` | `Syntax error on line N: check for missing brackets or typos.` |
   | `'window' is not defined` | `'window' isn't available in CodeQuest — you don't need it here.` |
   | `ReferenceError: x is not defined` | `'x' hasn't been created yet. Did you forget to declare it with 'let'?` |
   | `Maximum call stack size exceeded` | `Your code ran in circles too many times. Check for an infinite loop.` |
   | `Cannot read properties of undefined` | `Something went wrong — try checking your function arguments.` |
   | Unknown function | `'functionName' isn't one of the available functions for this level. Check the function list!` |
   | `var` or `const` used | `Use 'let' instead of 'var' or 'const' to create variables in CodeQuest.` |

3. Do not expose the raw acorn parse error message to the player — only use it to extract the line number. The display message is always one of the controlled templates above.

4. Export `PlayerFriendlyError` and `formatErrorForDisplay` from `src/engine/interpreter/index.ts` for consumption by E-11 (lesson runner).
