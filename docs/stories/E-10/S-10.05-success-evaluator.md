## What

`successEvaluator.ts` implements semantic comparison between what the player drew and what the solution expects. Semantic comparison avoids brittle pixel-diff approaches: small numeric differences (e.g. a circle at 148 vs 150) are accepted within a configurable tolerance.

The evaluator checks the player's `callLog` against `phase3.canvas.solution.calls` from the lesson JSON. Calls must appear in the same order. String arguments (colors) must match case-insensitively. Numeric arguments must be within `tolerance` of the expected value. If the player makes *extra* calls beyond the solution set, those are ignored — the solution defines the minimum required output, not a strict equality.

## Tasks

1. Create `src/renderer/successEvaluator.ts`:
   ```ts
   import type { RendererCall } from './rendererAPI';

   export interface SolutionCall {
     fn: string;
     args: (string | number)[];
   }

   export function evaluateSuccess(
     playerCalls: RendererCall[],
     solution: SolutionCall[],
     tolerance: number,
   ): boolean {
     if (playerCalls.length < solution.length) return false;

     let playerIndex = 0;

     for (const expected of solution) {
       // Advance through player calls to find the next matching call
       while (playerIndex < playerCalls.length) {
         if (callMatches(playerCalls[playerIndex], expected, tolerance)) {
           playerIndex++;
           break;
         }
         playerIndex++;
       }
       if (playerIndex > playerCalls.length) return false;
     }

     return true;
   }

   function callMatches(
     actual: RendererCall,
     expected: SolutionCall,
     tolerance: number,
   ): boolean {
     if (actual.fn !== expected.fn) return false;
     if (actual.args.length !== expected.args.length) return false;

     for (let i = 0; i < expected.args.length; i++) {
       const exp = expected.args[i];
       const act = actual.args[i];
       if (typeof exp === 'string' && typeof act === 'string') {
         if (exp.toLowerCase() !== act.toLowerCase()) return false;
       } else if (typeof exp === 'number' && typeof act === 'number') {
         if (Math.abs(act - exp) > tolerance) return false;
       } else {
         // Type mismatch (e.g. player passed string where number expected)
         return false;
       }
     }

     return true;
   }
   ```

2. Note on ordering: the current implementation uses a sequential scan — solution calls must appear in order relative to each other in the player's log, but extra calls between solution calls are allowed. If "order matters strictly" is desired (no extra calls between), change the loop to strict index matching. Document the chosen behaviour clearly.

3. Write unit tests in `src/renderer/successEvaluator.test.ts`:
   - Exact match returns `true`
   - Numeric arg within tolerance returns `true`
   - Numeric arg outside tolerance returns `false`
   - String arg case mismatch returns `false` (e.g. `"Red"` vs `"red"` — should pass; `"blue"` vs `"red"` — should fail)
   - Player has fewer calls than solution returns `false`
   - Extra player calls beyond solution are allowed (returns `true`)
   - Wrong function name returns `false`
