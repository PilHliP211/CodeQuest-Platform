# S-10.03: API Whitelist Enforcement

**Epic:** E-10
**Size:** XS
**Dependencies:** S-10.02 (full `RendererAPI`), S-09.02 (sandbox — understanding how the interpreter scope is built)

## Skills to Load

- `security-rules` — whitelist pattern; never expose more than the lesson allows
- `typescript-standards` — typed `Record` for interpreter scope injection
- `content-pack-system` — `availableFunctions` lives in `phase3.canvas.availableFunctions`

## What

The full `RendererAPI` has five drawing functions. Each lesson declares only the subset the player should use in `phase3.canvas.availableFunctions`. This story adds a filtering step: given the full `RendererAPI` object and the lesson's `availableFunctions` array, produce a restricted scope object containing only the allowed functions.

This filter is applied at the point where the interpreter scope is built — the resulting object is what gets injected into the interpreter execution context. Functions that exist in `RendererAPI` but are not listed in `availableFunctions` are simply absent from the scope. If the player tries to call one, the interpreter's own unknown-function check (E-09 sandbox) returns the friendly "That function isn't available here" error.

## Tasks

1. Add to `src/renderer/rendererAPI.ts` (or a new `src/renderer/rendererScope.ts` — author's choice):
   ```ts
   import type { AvailableFunction } from '@/types/content';

   export function buildRendererScope(
     api: RendererAPI,
     availableFunctions: AvailableFunction[],
   ): Record<string, (...args: unknown[]) => void> {
     const allowed = new Set(availableFunctions.map((f) => f.name));
     const scope: Record<string, (...args: unknown[]) => void> = {};

     const fnNames: (keyof RendererAPI)[] = [
       'fillBackground',
       'drawRect',
       'drawCircle',
       'drawStripe',
       'drawTriangle',
     ];

     for (const name of fnNames) {
       if (allowed.has(name)) {
         scope[name] = api[name] as (...args: unknown[]) => void;
       }
     }

     return scope;
   }
   ```

2. The scope object passed to the interpreter must **not** include `getCallLog` or `clearCallLog` — those are internal methods, not player-callable functions. The iteration above over a fixed list of drawing function names enforces this.

3. Wire this into the lesson runner (E-11) or the Phase 3 run handler: when the player clicks "Run", call `buildRendererScope(api, lesson.phase3.canvas.availableFunctions)` and pass the result to the interpreter as its initial scope.

## Done When

- [ ] `buildRendererScope` exported and typed (no `any`)
- [ ] Only names present in `availableFunctions` appear in the returned scope object
- [ ] `getCallLog` and `clearCallLog` are never in the returned scope
- [ ] A lesson with `availableFunctions: [{ name: 'fillBackground' }, { name: 'drawCircle' }]` produces a scope with exactly those two keys
- [ ] `npm run typecheck` and `npm run lint` pass
