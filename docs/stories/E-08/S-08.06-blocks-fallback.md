# S-08.06: "Show Me in Blocks" Fallback After 3 Failed Attempts

**Epic:** E-08
**Size:** XS
**Dependencies:** S-08.03 (`EditorToggle`), S-11.03 (attempt counter from lesson runner)

- `component-tests` — Testing Library behavior tests for the component added here

## What

After 3 failed run attempts in Phase 3, a "Need help? Try blocks →" button appears. Clicking it switches the `EditorToggle` to the `'blocks'` view. The blocks view in Phase 3 is always read-only — it shows the block structure corresponding to the current editor code but cannot be executed from. The player must switch back to the syntax editor to run code.

This story wires the attempt count from the lesson runner (E-11) to the toggle UI.

## Tasks

1. In the Phase 3 lesson screen (to be built in E-11), read `attemptCount` from `LessonContext`:
   ```tsx
   const { lessonState } = useLessonRunner();
   const showFallback = lessonState.attemptCount >= 3;
   ```

2. Render the fallback button conditionally:
   ```tsx
   {showFallback && (
     <button
       type="button"
       onClick={() => setActiveView('blocks')}
       className="font-pixel text-xs text-yellow-300 underline"
     >
       Need help? Try blocks →
     </button>
   )}
   ```

3. When the blocks view is active in Phase 3, `BlockEditor` receives `readOnly={true}`. The "Run ▶" button is not shown. A small label reads: "Blocks view is read-only in Phase 3. Switch back to code to run." This label appears only in Phase 3, not Phase 1/2.

4. The `EditorToggle` always shows both tabs in Phase 3. The fallback button is an *additional* affordance below the editor — it does not replace the tab. Once the player switches to blocks voluntarily or via the fallback, they can switch back freely.
